Project Overview

A simple, peaceful treasury management application built with SvelteKit and Firebase. Designed for tracking income and expenses across multiple treasuries with a warm, approachable interface.

Frontend

SvelteKit with TypeScript for a reactive, type-safe UI. Tailwind CSS provides the warm terracotta and sage color palette.

Backend

Firebase handles authentication, Firestore database, and real-time updates. Server-side rendering compatible.

Features

Multi-treasury support, transaction tracking, category management, balance calculations, and responsive mobile-first design.

Authentication Flow

Firebase Auth provides email/password authentication. The auth.ts module exports a Svelte store that reacts to auth state changes, enabling reactive UI updates across the application.

Data Access Pattern

All Firestore operations are encapsulated in treasury.ts. This provides a clean API for components while handling Firebase-specific concerns like timestamps and document references.

Known Issues

Current limitations and technical debt to be aware of.

Medium

Category Type Ambiguity

Custom categories added via addCategory() don't store whether they're income or expense categories. The UI merges defaults with custom categories for both types, which could lead to categorization confusion.

Medium

No Real-time Updates

Data is fetched on page load and after explicit actions. Multiple users viewing the same treasury won't see each other's changes without refreshing.

Low

Missing Input Validation

Amount inputs rely on HTML5 number type but don't have explicit validation for negative numbers or extremely large values in the business logic.

Security

No Firestore Security Rules Documented

The application assumes client-side filtering by userId, but Firestore security rules should be configured to enforce user isolation at the database level.

Low

SSR Compatibility Gaps

While firebase.ts has SSR guards, some reactive statements in pages may still attempt to access browser-only APIs during server rendering.

Improvement Opportunities

Potential enhancements for future development.

Real-time Sync High
Use Firestore onSnapshot listeners to enable live updates across clients. This would allow multiple treasurers to collaborate simultaneously.
Export Functionality Medium
Add CSV/PDF export for treasuries and transactions. Useful for reporting and backup purposes.
Transaction Search & Filter Medium
Add search by note/category and date range filtering for transaction history.
Recurring Transactions Medium
Support scheduled recurring transactions for regular expenses like rent or monthly donations.
Budget Planning Low
Add budget categories with spending limits and visual progress indicators.
Offline Support High
Enable Firestore offline persistence so users can add transactions without connectivity.
Data Visualization Low
Add charts for spending trends over time using Chart.js or similar library.
Multi-user Treasury Access High
Allow multiple users to manage the same treasury with role-based permissions (viewer, editor, admin).
